# Screenshot Tool

A spec-driven CLI tool that builds, launches, and navigates an app to a desired screen/state via deep link, then captures a screenshot.

> [!NOTE]
> This is a **specification-only** repository. There is no reference implementation — just a spec. You bring your coding agent and build it in your codebase, in your language, for your platform.

## The Problem

Coding agents can write UI code, but they can't see the result. They generate layouts, styles, and states — then move on, hoping it looks right. Without visual feedback, mistakes pile up: broken layouts, missing states, overlapping elements.

## The Solution

Give your coding agent a screenshot tool. Once built from this spec, the agent can:

1. **Generate or modify UI code**
2. **Run `take_screenshot`** to capture the rendered screen
3. **Inspect the screenshot** to verify the result matches the intent
4. **Iterate** — fix issues and re-capture until the UI is correct

The tool uses deep links and a screen registry (`screens.yaml`) so the agent can navigate to any screen in any state — default, loading, error, empty — and capture exactly what the user would see.

This closes the feedback loop: the agent doesn't just write UI code, it validates it.

## Why Spec-Driven?

Every project has different build systems, simulators, deep link schemas, and screenshot tooling. A single implementation can't cover all of them. A spec can.

- **Language-agnostic** — Implement in Bash, Python, Swift, Kotlin, TypeScript, or anything else
- **Platform-agnostic** — The core interface is identical; only execution mechanics change per platform
- **Agent-friendly** — Coding agents excel at turning precise specs into working code
- **Your codebase, your rules** — The implementation lives in your repo, not ours

## What's in the Spec

[`SPEC.md`](SPEC.md) is a complete, self-contained specification covering:

- **Interface** — CLI arguments, flags, exit codes
- **Deep Link Contract** — `screens.yaml` registry for screens, states, and params
- **Platform Support** — Platform-specific parameters alongside a universal core interface
- **Behavior** — Build → Launch → Navigate → Settle → Screenshot → Save → Teardown → Exit
- **Deliverables** — `SKILL.md` companion file for coding agent integration
- **Test Cases** — Comprehensive checklist covering navigation, build/launch, output, correctness, timeout, teardown, platform params, and help/discovery

## Getting Started

Point your coding agent (Claude Code, Cursor, Copilot, Codex, etc.) at the spec:

```
Build a screenshot tool following the specification in SPEC.md.
Target platform: [iOS / Android / Web / Desktop].
```

The agent reads the spec and produces a working implementation tailored to your project.

## Examples

Both examples below were generated by Claude Code from a single prompt pointing at `SPEC.md`. Each directory contains the complete tool, agent skill guide, and screen registry.

### macOS Desktop App (SwiftUI + Xcode)

[`examples/macos-desktop/`](examples/macos-desktop/) — Bash, `xcodebuild`, native `NSView.cacheDisplay` capture

| File | Description |
|------|-------------|
| [`take_screenshot`](examples/macos-desktop/take_screenshot) | 439-line Bash script — builds, launches, captures via native APIs |
| [`SKILL.md`](examples/macos-desktop/SKILL.md) | Coding agent usage guide with batch mode tips |
| [`screens.yaml`](examples/macos-desktop/screens.yaml) | 2 screens, 5 states, parameter definitions |

```bash
./take_screenshot --url "cmuitestapp://properties-panel"
./take_screenshot --url "cmuitestapp://properties-panel?state=error&error_code=404"
./take_screenshot --url "cmuitestapp://connection-detail?state=default&connection_id=42"
```

### Web App (React + Playwright)

[`examples/web-react/`](examples/web-react/) — TypeScript, Playwright headless browser, API mocking, device emulation

| File | Description |
|------|-------------|
| [`take_screenshot.ts`](examples/web-react/take_screenshot.ts) | 936-line TypeScript script — builds, starts dev server, mocks APIs, captures with Playwright |
| [`SKILL.md`](examples/web-react/SKILL.md) | Coding agent usage guide with device viewport examples |
| [`screens.yaml`](examples/web-react/screens.yaml) | 10 screens, 30+ states, auth-aware routing |

```bash
npx tsx take_screenshot.ts --url "myapp://dashboard"
npx tsx take_screenshot.ts --url "myapp://login?state=error"
npx tsx take_screenshot.ts --url "myapp://slot-details?id=slot-1" --device mobile
```
